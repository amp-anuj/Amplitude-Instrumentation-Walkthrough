The Amplitude Platform

Amplitude is a digital analytics and product intelligence platform. Teams use it to understand user behavior, run experiments, guide users through key moments, and replay exactly what users experienced.

📊
Analytics
Track events, identify users, and analyze behavior. Understand how users navigate your product, where they convert, and what drives retention. The foundation all other products build on.
🧪
Experiment
Run A/B tests and feature flags with statistical confidence. Ship features to targeted groups, measure impact on key metrics, and make data-driven decisions about what to roll out.
🗺️
Guides & Surveys
Deliver contextual in-app guidance and collect feedback at the right moment. Trigger guides, tours, and surveys based on user behavior — no code deploys required after initial SDK setup.
🎥
Session Replay
See exactly what users saw. Combine behavioral analytics with visual session recordings — jump from a funnel drop-off directly to the replays of affected users to diagnose friction.
How the products work together
Analytics is the foundation — all other SDKs initialize on top of it and share the same device_id, session, and user identity
Experiment targeting uses Analytics user properties and cohorts — no separate identity setup needed with initializeWithAmplitudeAnalytics
Guides & Surveys fire in real time on Analytics events — event-based triggers work because the SDK receives the same event stream
Session Replay links to Analytics events via a shared Session Replay ID — jump from a funnel step directly to the replay of affected users
Unified SDK. Starting fresh? @amplitude/unified (npm) or the Unified Script (CDN) initializes all four products in a single call — one API key, one SDK version to manage.

Analytics SDK — Overview

The Browser SDK

A client-side JavaScript SDK for capturing user events and properties. Works in any browser — no build tools or bundler required when loaded from CDN.

📡
Events
Actions users take in your product. Every amplitude.track() call sends an event to Amplitude with properties describing context.
👤
User Identity
Who is performing the action. Set with setUserId() and enriched with user properties via identify().
🔄
Sessions
Automatically tracked. A session is a period of continuous activity. Amplitude uses sessions to understand engagement patterns and retention.
🔌
Plugins
Extend SDK behavior — Session Replay, Experiment, and Guides & Surveys all integrate as plugins on top of the Browser SDK.
The Amplitude data model
📌 Event
An action a user takes. Has an event type (name) and event properties (context about that action).

amplitude.track('Song Played', { genre: 'jazz', duration: 180 })
👤 User Properties
Attributes that describe the user — plan, role, country. Persist across sessions. Set via amplitude.identify().

Used for segmentation, experiment targeting, and cohort membership.
🔑 User Identity
User ID — your app's ID for the logged-in user.
Device ID — SDK-generated, persists per browser/device. Required for anonymous users.
Amplitude ID — Amplitude's internal merge key; created on first event.
Unified SDK — If starting fresh, consider @amplitude/unified which bundles Analytics, Experiment, Session Replay, and Guides & Surveys in a single package. Load once, configure all products.

Autocapture vs Precision Tracking

These two approaches aren't either/or — the recommended approach combines both. Autocapture handles broad discovery; precision tracking handles what matters most.

✦ Autocapture
Enabled by default. Automatically tracks page views, sessions, form interactions, file downloads, and attribution. Enable elementInteractions separately for click-level data.
Good for
Exploratory analysis · New features · Campaign monitoring · Site redesigns
Precision Tracking
Custom amplitude.track() calls. You control the event name and every property. Amplitude is the source of truth for critical business events like subscriptions and purchases.
Good for
Purchase flows · Custom properties · Server-verified events · Subscription state
Autocapture by tier
Tier 1 — Quick Insight
✓ Page views
✓ Sessions (start/end)
✓ Attribution (UTM, referrer, click IDs)
✓ Form interactions
On by default
Tier 2 — User Clarity
✓ Element clicks (all)
✓ File downloads
✓ Rage clicks & dead clicks
✓ Frustration signals
elementInteractions: true
Tier 3 — Engineering Context
✓ Network failures
✓ SPA navigation (history API)
✓ Visual Labeling selectors
✓ Console errors (opt-in)
Individual toggles
Visual Labeling: Autocapture collects raw element interactions. Use Amplitude's Visual Labeling UI to transform them into named, meaningful events — without any additional code changes.
Volume note: Autocapture events count toward event volume. elementInteractions in particular can generate high volume — estimate DAU × average page interactions before enabling broadly.

Identify users

Amplitude tracks every user by a device_id (anonymous) and optionally a user_id (authenticated). User properties power targeting in Experiment and Guides & Surveys.

// Call after user logs in
amplitude.setUserId('user_demo_001');

// Attach properties for targeting & segmentation
const identifyEvent = new amplitude.Identify();
identifyEvent.set('plan', 'growth');
identifyEvent.set('role', 'analyst');
amplitude.identify(identifyEvent);
User properties matter beyond Analytics. Any property set here flows into Experiment's variant targeting and Guides & Surveys audience filters — e.g. show a guide only to plan = 'trial' users.

Event vs User Properties
Event Properties
Context for this specific action. Only attached to the event they're tracked with. Not visible on other events.
button_id · page · search_query · payment_method
User Properties
Context for who the user is. Persists across all subsequent events until updated. Use for plan, role, cohort membership.
plan · role · country · subscription_tier
Naming conventions
✓ Consistent
Button Clicked · Page Viewed · Purchase Completed
✗ Avoid
buttonClick · page_view · PURCHASE_COMPLETE
Pick one convention (Title Case, snake_case, or camelCase) and apply it everywhere. Mixed casing creates duplicate events in Amplitude.
A/B test data should be sent as user properties — one property per experiment, value = variant name. This lets you segment any chart by experiment variant without extra event instrumentation.

Analytics SDK — Implementation

Groups & group properties

Groups enable account-level analytics — query by organization, company, or any entity rather than individual users. Requires the Accounts add-on. Use setGroup() to associate users and groupIdentify() to set properties on the group itself.

setGroup — assign user to group 
// Associate the current user with a group
// group_type: the category (e.g. 'orgId', 'company', 'plan')
// group_name: the specific value (e.g. '12345', 'Acme Corp')
amplitude.setGroup('orgId', '12345');

// A user can be in multiple groups of the same type:
amplitude.setGroup('orgId', ['12345', '67890']);
groupIdentify — set group properties 
// Set properties ON the group (not the user)
// These persist to all future events from members of this group
const groupEvent = new amplitude.Identify();
groupEvent.set('plan', 'enterprise');
groupEvent.set('arr', 120000);
groupEvent.set('csm', 'taylor@amplitude.com');
amplitude.groupIdentify('orgId', '12345', groupEvent);
event-level group (track call) 
// Attach a group to a single event without persisting to user
// Useful when the user's group context changes per-event
amplitude.track({
  event_type: 'Feature Used',
  event_properties: { feature: 'dashboard' },
  groups: { orgId: '12345' }  // event-level only
});
Try it
Three group methods
setGroup()
Assigns the user to a group. Also sets groupType:groupName as a user property — overwrites any existing value for that group type.
groupIdentify()
Sets properties on the group itself (plan, ARR, CSM, etc). These persist and appear on all events from any member of the group going forward.
track({ groups })
Attaches a group to a single event only. Does not update the user's group membership — use setGroup() for persistent membership.
Accounts add-on required. Groups and group-level reporting ("Count by Distinct on orgId") require the Accounts add-on. Without it, setGroup() still sets a user property but group-level charts won't be available.
B2B use cases
group_typeorgId · company · workspace · account
group propsplan · arr · csm · industry · tier · seats
queriesCount by Distinct, DAU by org, retention by plan tier
groupIdentify only affects future events. Properties set on a group do not retroactively update historical events from group members.

Experiment SDK — Overview

Feature Flags vs Experiments

From an application code perspective there is no difference — both use variant(flag_key). The distinction is in how you use the results.

A/B Experiments
Make statistically sound decisions about rolling out features. Test whether a new experience improves a metric or doesn't negatively impact key metrics.
Use for
Data-driven rollout decisions · Hypothesis validation · Impact measurement
Feature Flags
Gate or release features to targeted groups. Manage rollout of an experiment after it reaches significance. Kill switch for emergencies.
Use for
Beta access · Trunk-based dev · Controlled releases · Kill switches
Important: Analysis in Amplitude Experiment is always done by amplitude_id. For experiments (not flags), bucket users by user ID to ensure consistent analysis across devices.

Experiment SDK — Overview

Deployments

A deployment is a collection of flags and experiments tied to a single Amplitude Analytics project. Your application uses the deployment key to initialize the SDK.

Key facts about deployments
One deployment → one Analytics project. Assignment events go to that project.
Multiple deployments can share the same Analytics project.
A flag/experiment can only belong to deployments sharing a common Analytics project.
Deployments are either client-side or server-side — use client-side for Browser SDK.
The deployment key is used to initialize the Experiment SDK (not the Analytics API key).
Where to find your deployment key: Amplitude → Experiment → Deployments → select your deployment → copy the key. It starts with client- for client-side deployments.

The two providers

The Experiment SDK needs two things wired up at initialization: a User Provider that tells it who the current user is, and an Exposure Tracking Provider that tells it where to send exposure events. How these are configured depends on your analytics setup.

The Experiment SDK needs identity and a destination for exposure events. With initializeWithAmplitudeAnalytics, both are wired automatically from your existing Analytics SDK — no manual implementation needed.

😃
ExperimentUserProvider
Implements getUser(). Called on every fetch() to identify the current user for variant assignment. Returns { user_id, device_id } — at least one is required.
Required
⚡

Implements track(exposure). Called automatically when variant() is invoked. Sends a $exposure event to your analytics provider so Amplitude can compute experiment results.

✦

Exposure & outcome tracking

Exposure events tell Amplitude which users actually saw each variant. Without them, you can't compute experiment results.

Session Replay — Overview

      
Sample rate & privacy

      
Not every session needs to be recorded. Set a sample rate to balance coverage against storage costs, and mask sensitive content before it ever leaves the browser.


      

        

          
Sample rate

          
A value from 0 to 1 controlling the percentage of sessions captured. sampleRate: 1 records everything. 0.1 records 10%. Start high during testing, lower in production to control volume.

        

        

          
Session Replay ID

          
When a session is sampled in, the SDK attaches [Amplitude] Session Replay ID as an event property on all events in that session. This is how replays link to your analytics data.

        

        

          
Default masking

          
Text inputs, passwords, and elements with the .amp-block class are masked by default. Images can be blocked. Masking happens client-side — data never leaves the browser.

        

        

          
CSS selector blocking

          
Use privacyConfig.blockSelector to block entire elements by class, ID, or any CSS selector — useful for PII, payment fields, or health data sections.

        

      


      

        
Out-of-the-box masking levels

        

          

            
🔒 Conservative

            
Masks all text content and images across the entire page. Maximum privacy — minimal visual fidelity in replays. Good for heavily regulated industries (healthcare, finance).

          

          

            
🟡 Moderate

            
Masks inputs and form fields. Other text content is visible. Default level. Balances privacy with useful replay fidelity for most products.

          

          

            
🟢 Liberal

            
Only masks elements explicitly marked with .amp-block or matched by your blockSelector. Maximum visual clarity — use only for internal tools or non-sensitive pages.

          

        

      


      

        Remote Sample Rate — admins can override the SDK's sampleRate from the Amplitude UI without a code deploy. Found under Settings → Session Replay. Changes take a few minutes to propagate. The remote value always wins over the SDK value.
      


      

        Only sampled sessions have a Replay ID. Events from sessions that aren't sampled in will not have the [Amplitude] Session Replay ID property and won't appear in the replay viewer.
      


      

      

        
sampleRate defaults to 0. If you forget to set it, nothing will be recorded — no error is thrown. Always set it explicitly. Use sampleRate: 1 during development and QA, then lower it for production.

      

      

        

          
Sample rate guidance

          
Starting point20–50% is a good baseline. Use: quota ÷ monthly sessions to find your ceiling.

          
Dev / QA1.0 — capture everything to validate implementation

          
Production0.01–0.1 typical (1–10%). Balance coverage vs. storage.

          
FunnelsConsider higher rate on key conversion pages where replay adds most value.

        

        

          
What Session Replay captures

          

            ✓ DOM mutations, clicks, scrolls, form interactions

            ✓ Network errors and console errors (optional)

            ✗ Not a screen recording — no camera, microphone, or native app UI

            ✗ iframes from cross-origin domains are blocked
          

        

        

          Session Replay uses rrweb to serialize DOM state and mutations — not pixel-based video capture. Replay data is stored separately from event data and retained for 90 days by default (configurable to 30).
